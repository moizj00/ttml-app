
The letter generation pipeline has two categories of problems that need to be fixed together:
Backend/Pipeline: The pipeline's status mutations are incomplete and fragile. The drafting status is never set in the happy path — the subscriber sees "Researching" for the entire pipeline duration, then a sudden jump to "Draft Ready". The direct (non-n8n) pipeline path doesn't even set researching. The subscriber gets no email when the pipeline completes normally (only in the degraded fallback path). Version pointer updates aren't guarded before status transitions, creating a race where Supabase realtime fires "Draft Ready" before the draft version is actually accessible. On complete failure, the subscriber gets no notification at all — only admins are alerted.
Frontend/Design: Three inconsistent progress representations (7-step dashboard stepper, 4-step detail bar + 6-step vertical timeline, no progress on My Letters). Animation hooks (useStepTransition, useProgressFill) exist but are never wired in. The dashboard stepper forces 544px minimum width causing horizontal scroll on mobile.

How the Pipeline Currently Works (Plain English)
When a subscriber submits a letter, here's what happens today:

Submit — The letter request is saved to the database with status submitted. A job is queued for the worker.
Worker picks it up — The worker calls runFullPipeline(). If using the n8n path, it sets status to researching and fires off to n8n. If using the direct 4-stage path (the default/primary), it does NOT update status at all — it stays at submitted.
Research stage — Calls Perplexity (or fallback providers) to research applicable laws. No status update after this completes.
Drafting stage — Calls Claude/OpenAI to draft the letter using research results. No status update.
Assembly & Vetting — Runs a quality check loop. If critical issues found, flags them. No status update.
Finalization — finalizeLetterAfterVetting() creates a letter version, updates version pointers, then sets status to generated_locked. This is the FIRST status update the subscriber sees since submitted (or researching on n8n path). No email is sent to the subscriber.
Payment gate — Subscriber must pay to unlock attorney review. Status changes to pending_review.
Attorney review — Attorney claims, reviews, approves/rejects/requests changes.
Client approval — Subscriber approves, requests revision, or declines.

What goes wrong:

Steps 2-5 are invisible to the subscriber — no status updates fire during the bulk of the work
If the pipeline fails, it retries up to 4 times with exponential backoff (10s, 20s, 40s)
If all retries fail, bestEffortFallback() tries to salvage a degraded draft. If that works, subscriber gets an email. If it doesn't, status goes to pipeline_failed and only admins are notified — the subscriber is left in the dark.
The force: true flag on some status updates completely bypasses the state machine validation


Part 1: Backend Pipeline Robustness
1A. Add researching status in the direct pipeline path
File: server/pipeline/orchestrator.ts (~line 268)
The direct 4-stage pipeline (the primary/default path) never sets researching. Only the n8n path does (line 147). Fix: add await updateLetterStatus(letterId, "researching") after creating the workflow job, before the research stage runs.
1B. Add drafting status in the happy path
File: server/pipeline/orchestrator.ts (~line 335)
Currently: runResearchStage() → runDraftingStage() → runAssemblyVettingLoop() → finalizeLetterAfterVetting() with NO status updates between them.
Fix: Add await updateLetterStatus(letterId, "drafting") before runDraftingStage(). Valid per ALLOWED_TRANSITIONS: researching → drafting.
After this fix, the happy path status flow becomes:
submitted → researching → drafting → generated_locked
Each transition fires a Supabase realtime event, so the subscriber sees real-time progress.
1C. Guard version creation before status transition
File: server/pipeline/vetting.ts (lines 881-911 in finalizeLetterAfterVetting)
Currently: createLetterVersion → updateLetterVersionPointers → updateLetterStatus("generated_locked"). If version creation returns a zero/null ID, or if pointer update fails, the status still advances. The Supabase realtime broadcasts "Draft Ready" before the version is accessible.
Fix: Validate versionId is truthy before proceeding. If it's 0/null, throw an error so the pipeline retries rather than leaving the letter in a broken state.
typescriptconst version = await createLetterVersion({ ... });
const versionId = (version as any)?.insertId ?? 0;
if (!versionId) throw new Error(`Failed to create letter version for letter #${letterId}`);

await updateLetterVersionPointers(letterId, { currentAiDraftVersionId: versionId });
// Only now that the version is persisted and pointed to, update status
await updateLetterStatus(letterId, "generated_locked");
1D. Add subscriber email on normal pipeline completion
File: server/pipeline/vetting.ts (in finalizeLetterAfterVetting, after line 925)
Currently sendLetterReadyEmail() is only called in bestEffortFallback() (degraded path). Normal pipeline completion via finalizeLetterAfterVetting never emails the subscriber.
Fix: After the status update and review action log, send the "Your letter draft is ready" email to the subscriber. Fire-and-forget (.catch()) like the existing pattern in bestEffortFallback.
1E. Add subscriber notification on complete pipeline failure
File: server/worker.ts (~line 154-175, after bestEffortFallback fails)
Currently when the pipeline fails completely (all retries + fallback exhausted), only admins get email alerts and in-app notifications. The subscriber sees pipeline_failed status but gets no email explaining what happened.
Fix: Send a sendStatusUpdateEmail() to the subscriber informing them that their letter encountered an issue and our team has been notified. Include a CTA to contact support or retry.
1F. Log status transitions for audit trail
File: server/pipeline/orchestrator.ts
The researching and drafting transitions should be logged via logReviewAction() for audit consistency. The finalizeLetterAfterVetting already logs the drafting → generated_locked transition — add similar logging for submitted → researching and researching → drafting.
1G. Add pipeline lock to processRetryFromStage
File: server/worker.ts (in processRetryFromStage, ~line 199)
Critical race condition: processRetryFromStage does NOT acquire the pipeline lock before running. This means a stage-aware retry can execute concurrently with a full pipeline run or another retry, because only processRunPipeline calls acquirePipelineLock().
Fix: Add acquirePipelineLock(letterId) at the start of processRetryFromStage and releasePipelineLock(letterId) in a finally block, matching the pattern in processRunPipeline.
1H. Stop status oscillation during retries
File: server/worker.ts (~lines 76-78 in retry loop)
Currently the worker forcibly resets status to submitted before every retry attempt:
typescriptif (!current || current.status !== "submitted") {
  await updateLetterStatus(letterId, "submitted", { force: true });
}
This causes the subscriber to see status bounce backward: researching → submitted → researching → submitted → ... on each retry. With Supabase realtime, this fires confusing status change events and toast notifications ("Your request has been received" repeatedly).
Fix: Only reset to submitted on the first retry (attempt 1), not on every attempt. On subsequent retries, check if the letter is in a retryable state (submitted, researching, drafting, pipeline_failed) and only force-reset if it's stuck in a non-retryable state. Better: skip the force-reset entirely if the pipeline's own error handler (orchestrator.ts line 424) already reverted to submitted.
1I. Guard autoAdvanceIfPreviouslyUnlocked against race conditions
File: server/pipeline/orchestrator.ts (~lines 672-757 in autoAdvanceIfPreviouslyUnlocked)
This function uses a SELECT-then-UPDATE pattern:
typescriptconst current = await getLetterRequestById(letterId);
if (current.status === "generated_locked") {
  await updateLetterStatus(letterId, "pending_review");
}
Between the SELECT and UPDATE, another process (retry pipeline, payment handler) could change the status. The updateLetterStatus call would then fail because pending_review is only valid from generated_locked.
Fix: Remove the pre-check SELECT and just call updateLetterStatus(letterId, "pending_review") directly. The function already validates the transition — if the letter isn't in generated_locked, the update will be a no-op (already at target) or throw (invalid transition). Wrap in try/catch and log the race instead of crashing.
1J. Prevent stale force: true bypass in bestEffortFallback
File: server/pipeline/orchestrator.ts (~line 590 in bestEffortFallback)
bestEffortFallback uses force: true to set status to generated_locked, bypassing the state machine entirely. This is dangerous because if the letter has already been advanced (e.g., subscriber paid during retry window), the force update would regress it back to generated_locked.
Fix: Before the force update, check current status. If it's already past generated_locked (e.g., pending_review, under_review, approved), skip the fallback entirely — the letter has already progressed beyond what the fallback would deliver.
typescriptconst current = await getLetterById(letterId);
const postLockStatuses = ["pending_review", "under_review", "approved", "needs_changes",
  "client_approval_pending", "client_revision_requested", "client_approved", "sent"];
if (current && postLockStatuses.includes(current.status)) {
  console.log(`[Pipeline] Letter #${letterId} already at ${current.status} — skipping bestEffortFallback`);
  return true; // Already progressed, no need for fallback
}
await updateLetterStatus(letterId, "generated_locked", { force: true });
