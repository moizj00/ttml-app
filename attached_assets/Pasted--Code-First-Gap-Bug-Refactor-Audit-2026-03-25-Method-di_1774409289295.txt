## Verified implementation facts from code

1. **Four roles are implemented in schema and routing** (`subscriber`, `employee`, `attorney`, `admin`).
2. **Affiliate mechanics exist**: employee codes, discount validation, commission ledger/payout flow, Stripe metadata enrichment.
3. **Review center is attorney+admin enabled**, not admin-only.
4. **Current product flow is draft-first/paywall-later** (`submit -> generated_locked -> unlock -> pending_review`).

---

## Gaps / errors vs requested target behavior

### A) Entitlement gate missing before AI pipeline starts (Critical)
`letters.submit` creates and pipelines requests without checking purchase/plan/quota entitlement first.

### B) Free unlock bypass still active (Critical)
`checkPaywallStatus` supports free-first-letter state and `freeUnlock` advances to `pending_review` without payment.

### C) Subscriber can access AI draft before admin approval (Critical)
Subscriber-facing version access allows `ai_draft` in generated-locked flow, not final-only visibility.

### D) Final review authority is not admin-only (Critical)
`review.approve/reject/requestChanges/saveEdit` are under `attorneyProcedure` (attorney can perform final actions when assigned).

### E) Submit taxonomy mismatch (High)
Canonical letter types in code do not match requested 5-type business taxonomy.

### F) Employee cannot view referred subscribers' letters (High)
Affiliate router includes code/earnings/commissions/payouts but no referred-letter listing/query path.

### G) Membership + annual pricing rules diverge (High)
Current plan model uses finite allowance fields (`lettersAllowed`) and does not implement requested membership per-letter charging model and annual-48 decrement semantics.

### H) Subscription discount persistence risk (Medium)
Referral Stripe coupon is created with `duration: "once"`, likely discounting only first subscription invoice.

### I) Route mismatch in employee commission notifications (Medium)
Webhook/emails point to `/employee/dashboard`, while client routes expose `/employee` (and aliases `/employee/referrals`, `/employee/earnings`).

### J) `getEmployees()` includes admins (Low / design bug)
DB helper named `getEmployees` returns both `employee` and `admin`, risking semantic mistakes in reporting and assignment code.

### K) Pricing flow mismatch for Single Letter purchase order (High)
Pricing page sends `single_letter` selection to `/submit` directly (not immediate checkout), conflicting with “purchase first, then submit”.

### L) Submit form state selection is hard-limited to California (High)
Subscriber intake jurisdiction selector currently provides only one state option (`CA`), despite multi-jurisdiction requirements.

### M) Employee dashboard route aliases are not separate areas (Medium)
`/employee`, `/employee/referrals`, and `/employee/earnings` all mount the same component; role has labels for multiple areas but no distinct views.

### N) Header does not surface employee discount code persistently (Medium)
Employee discount code is visible in affiliate dashboard, but there is no persistent header code display in shared app layout navigation/header.

---

## Refactor opportunities

1. **Central entitlement policy service** reused by submit, paywall, checkout, and webhook.
2. **Authorization policy split**: assignment logic separate from final-approval authority.
3. **Plan policy normalization** (single source for price, quota, decrement, discount persistence).
4. **Route constants module** shared by frontend and backend notification link generation.
5. **Employee area modularization** into separate referrals/earnings/assistance pages + backend endpoints.
6. **Contract tests for business rules** (entitlement, final-only visibility, admin-only approval, annual decrement, referral visibility, discount persistence).

---

## Suggested execution order
1. Enforce entitlement before submit + remove free unlock path.
2. Restrict subscriber content visibility to final-approved only.
3. Enforce admin-only final review actions.
4. Fix pricing/plan/quota policy + single-letter purchase order.
5. Implement employee referred-letter visibility and split employee routes.
6. Fix route constant mismatches and helper naming semantics.
