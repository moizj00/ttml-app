Supabase authentication flows
Email/password signup and login
The server/supabaseAuth.ts module implements a robust authentication workflow using Supabase Auth.
Key points:
Allowed signup roles. The module defines ALLOWED_SIGNUP_ROLES = ["subscriber", "employee"][2], so a new user can sign up only as a Subscriber or an Employee. The Attorney role is not self‑assignable and must be granted by an administrator.
Signup endpoint (/api/auth/signup).
The request must include an email, password and the desired role (subscriber or employee). If the role is not in ALLOWED_SIGNUP_ROLES, the request is rejected.
The server calls Supabase’s auth.signUp to create the user and sends a verification email. It also inserts the user into the user table with the requested role and sends a welcome email. Employees receive a discount code for subscriptions.
If the email is in the SUPER_ADMIN_EMAILS list or matches OWNER_OPEN_ID, the user is automatically promoted to the admin role.
Login endpoint (/api/auth/login).
The server calls auth.signInWithPassword with the email and password.
Before granting access, it checks that the email has been verified. Unverified users get an error instructing them to verify their email.
On successful login, the server syncs user data into the app database, sets a secure session cookie and returns the user’s role.
Role escalation. Admins can promote an existing user to the Attorney role by updating the user.role column in the database. The authentication API does not expose a direct endpoint for attorney signup; this is by design to ensure only trusted users become attorneys.
Google Sign‑In / OAuth
Google login is handled via a custom OAuth flow implemented in three endpoints:
/api/auth/google – initiates the OAuth flow. It generates a PKCE code verifier and challenge, stores them in a cookie and redirects to Supabase’s Google authorization URL. The role query parameter is sanitized so that only subscriber or employee are allowed[3]. Attempting to sign up as an attorney will be rejected.
/api/auth/callback – handles the callback when Supabase returns a code in the query string (server‑side flow). It exchanges the code using the previously stored PKCE verifier, obtains access and refresh tokens, signs the session cookie, and calls syncGoogleUser to upsert the user in your database. Roles are assigned similarly to email signup: requested role (subscriber/employee) or auto‑promoted to admin if the email matches the admin whitelist.
/api/auth/google/finalize – supports the implicit flow when the tokens are returned in the URL fragment (client‑side). It exchanges the tokens via Supabase’s getUser() and sets the session cookie.
Testing tips
To ensure that both email/password and Google signup/login flows work correctly for each role, consider the following:
Subscribers and employees: Try creating accounts for both roles using email/password and Google. Confirm that the role is stored correctly in your database and that you receive a verification email. After verifying the email, ensure you can log in and access the appropriate sections of the app. Employees should see any discount features intended for them.
Admin accounts: Add your admin emails to the SUPER_ADMIN_EMAILS array or set OWNER_OPEN_ID in your environment. Sign up with one of these emails (either via Google or email/password). Verify that the user is automatically assigned the admin role and has access to administrative routes (e.g., /admin).
Attorney role: Confirm that there is no way to self‑register as an attorney. Log in as an admin, promote a test user to attorney by updating their role in the database (via your admin UI or Supabase dashboard). Then log out and back in as that user to verify the attorney permissions work.
Error handling: Test invalid cases: using unsupported roles, unverified emails, expired verification links, and invalid login credentials. Ensure the API returns descriptive errors.
Conclusion
There is no separate ADMIN_APP_BASE_URL variable in your repository. The application uses a single APP_BASE_URL to build absolute URLs for both the main site and admin interface[1]. The authentication module fully implements signup and login flows for Subscribers and Employees using both email/password and Google OAuth, while attorneys must be promoted by an admin and cannot sign up directly[2]. By testing the flows as outlined above and removing unused variables, you can ensure that the platform operates smoothly across all roles and access points.
