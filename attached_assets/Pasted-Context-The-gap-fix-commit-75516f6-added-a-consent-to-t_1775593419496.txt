Context
The gap-fix commit (75516f6) added a consent-to-training check in server/pipeline/training-capture.ts (lines 90-111) that queries the database twice before capturing training data:

db.select({userId}).from(letterRequests).where(...).limit(1) — get letter owner
db.select({consentToTraining}).from(users).where(...).limit(1) — check consent

Any test that calls captureTrainingExample() now fails unless its mock DB supports the select().from().where().limit() chain AND returns a consenting user. The current test files (statusMachine.test.ts, sanitization.test.ts) are pure-logic tests that don't touch DB — they're fine. But a training-capture test suite needs proper mocking.
Additionally, the updateLetterStatus() force-transition audit (added in server/db/letters.ts) calls logReviewAction() internally. Any integration test for that function also needs the review-actions mock.
