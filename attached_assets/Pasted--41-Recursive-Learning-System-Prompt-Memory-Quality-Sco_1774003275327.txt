#41 - Recursive Learning System (Prompt Memory + Quality Scoring)
What & Why
Make the letter generation pipeline learn from itself. When attorneys approve, edit, reject, or request changes on AI-generated letters, those outcomes carry valuable lessons. This task captures that feedback in a structured pipeline_lessons table, injects relevant lessons into future drafting/assembly/vetting prompts, and tracks letter quality metrics over time so the platform can measure its own improvement.

Done looks like
A pipeline_lessons table stores structured lessons extracted from attorney review actions (approvals, rejections, edits, change requests), each tagged by letter type, jurisdiction, pipeline stage, and category.
A letter_quality_scores table stores per-letter metrics computed after final disposition: first-pass approval rate, number of revision cycles, edit distance between AI draft and final, vetting pass/fail counts, and attorney time-to-approve.
When the pipeline builds prompts for drafting (Stage 2), assembly (Stage 3), and vetting (Stage 4), it queries the lessons table for the top N most relevant lessons (filtered by letter type + jurisdiction) and injects them as a ## LESSONS FROM PAST REVIEWS block.
After an attorney approves, rejects, or requests changes, the system auto-extracts a lesson by comparing the AI draft vs. the final content and the attorney's notes, storing it as a new lesson row.
Admins can view, edit, activate/deactivate, and manually create lessons from a "Learning" admin page.
Admins can view a quality dashboard showing approval rates, average revision cycles, and trend lines over time, broken down by letter type.
Out of scope
AI-powered auto-extraction of lessons using an LLM (start with rule-based extraction from attorney notes; LLM summarization can come later).
"Golden letter" example library (future work building on this foundation).
Vetting rule auto-evolution (future work).
Anonymization of letters for the example library.
Tasks
Schema & types — Add pipeline_lessons table (id, letter_type, jurisdiction, pipeline_stage, category, lesson_text, source_letter_id, source_action, is_active, weight, created_by, created_at, updated_at) and letter_quality_scores table (id, letter_request_id, first_pass_approved, revision_count, vetting_pass_count, vetting_fail_count, attorney_edit_distance, time_to_first_review_ms, time_to_approval_ms, computed_score, created_at). Add corresponding Drizzle insert/select types to schema and shared types. Run db:push.

Lesson extraction hooks — In the attorney approve, reject, requestChanges, and saveEdit procedures, after the existing logic, extract a lesson from the attorney's notes and the diff context. Store it in pipeline_lessons with appropriate tags. For approve: compare AI draft content vs final approved content to detect what the attorney changed. For reject/requestChanges: extract the reason text as the lesson.

Quality score computation — After approve or reject, compute and store a quality score row. Revision count = number of letter_versions for that request. Vetting metrics = count of vetting workflow_jobs that passed vs failed. Attorney edit distance = simple word-level diff ratio between last AI draft and final approved content. Time metrics = timestamp differences between status transitions stored in review_actions.

Prompt injection — Modify the prompt builder functions (buildDraftingSystemPrompt, buildAssemblySystemPrompt, buildVettingSystemPrompt) to accept an optional lessons array. In the pipeline runner, before calling each stage, query pipeline_lessons for active lessons matching the current letter_type and jurisdiction, ordered by weight descending, limited to 10. Inject them as a formatted block in the system prompt.

Admin Learning page — Build an admin page at /admin/learning with two tabs: "Lessons" (CRUD table of pipeline_lessons with filters by letter type, jurisdiction, stage, active/inactive toggle) and "Quality" (dashboard showing approval rate over time, average revision count, average score by letter type — using simple chart components or stat cards).

Admin nav integration — Add "Learning" link to the admin sidebar nav in AppLayout.tsx and register the route in App.tsx.
