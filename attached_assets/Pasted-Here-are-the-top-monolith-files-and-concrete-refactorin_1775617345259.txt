Here are the **top monolith files** and concrete refactoring ideas for each:

---

### 1. `server/email.ts` (1,808 lines, 27 exports)
**Problem:** Every email template (welcome, letter ready, draft reminder, status update, admin alert, commission, etc.) lives in one file.

**Refactor:** Split into `server/emails/` directory:
- `server/emails/templates/` — one file per email (`letterReady.ts`, `statusUpdate.ts`, `adminAlert.ts`, `commission.ts`, `draftReminder.ts`, `welcome.ts`)
- `server/emails/send.ts` — shared `sendEmail()` + Resend client
- `server/emails/layouts.ts` — shared HTML layout/header/footer helpers
- `server/emails/index.ts` — re-export barrel

---

### 2. `server/supabaseAuth.ts` (1,744 lines)
**Problem:** Auth registration, login, logout, Google OAuth, token refresh, role resolution, super admin whitelist, 2FA, session management — all in one file.

**Refactor:** Split into `server/auth/`:
- `server/auth/email.ts` — register, login, password reset
- `server/auth/google.ts` — OAuth initiate, finalize, `syncGoogleUser`
- `server/auth/session.ts` — token refresh, session validation, logout
- `server/auth/roles.ts` — role resolution, super admin whitelist, promotion flow
- `server/auth/middleware.ts` — Express middleware for auth guards
- `server/auth/index.ts` — re-export

---

### 3. `server/routers/letters.ts` (1,326 lines)
**Problem:** All letter-related tRPC procedures (submit, detail, myLetters, clientApprove, clientDecline, clientRequestRevision, retryFromRejected, archive, admin operations) in one router.

**Refactor:** Split by actor/domain:
- `server/routers/letters/subscriber.ts` — submit, myLetters, detail, archive
- `server/routers/letters/clientApproval.ts` — clientApprove, clientDecline, clientRequestRevision
- `server/routers/letters/admin.ts` — admin-only letter operations, force transitions
- `server/routers/letters/index.ts` — merge sub-routers

---

### 4. `client/src/pages/subscriber/LetterDetail.tsx` (1,288 lines)
**Problem:** Letter detail page with inline `ClientApprovalBlock` (200 lines), `RejectionRetryBlock` (70 lines), send dialog, copy-to-clipboard, and the full page layout.

**Refactor:** Extract sub-components:
- `components/letters/ClientApprovalBlock.tsx`
- `components/letters/RejectionRetryBlock.tsx`
- `components/letters/SendLetterDialog.tsx`
- `components/letters/LetterContentDisplay.tsx`
- Keep `LetterDetail.tsx` as a thin orchestrator composing these

---

### 5. `server/stripeWebhook.ts` (843 lines)
**Problem:** One massive `stripeWebhookHandler` with a switch/case over every Stripe event type, each with its own business logic inline.

**Refactor:** Split into event handlers:
- `server/stripe/webhookRouter.ts` — signature verification + event dispatch
- `server/stripe/handlers/checkoutCompleted.ts`
- `server/stripe/handlers/subscriptionDeleted.ts`
- `server/stripe/handlers/invoiceFailed.ts`
- `server/stripe/handlers/commission.ts`
- Each handler is a pure function: `(event, deps) => Promise<result>`

---

### 6. `server/pipeline/vetting.ts` (1,070 lines) + `orchestrator.ts` (980 lines)
**Problem:** Vetting has assembly loop + finalization + admin alerts all inline. Orchestrator has n8n path + direct path + bestEffortFallback + autoAdvance + retry all in one file.

**Refactor:**
- `server/pipeline/finalize.ts` — extract `finalizeLetterAfterVetting` + subscriber/admin notifications
- `server/pipeline/fallback.ts` — extract `bestEffortFallback` + degraded draft logic
- `server/pipeline/autoAdvance.ts` — extract `autoAdvanceIfPreviouslyUnlocked`
- Keep `orchestrator.ts` as the thin `runFullPipeline` coordinator

---

### 7. `shared/types.ts` (1,069 lines, 54 exports)
**Problem:** Status config, letter types, transition rules, intake types, pipeline types, pricing types, UI config — all in one file.

**Refactor:** Split into `shared/`:
- `shared/status.ts` — `ALLOWED_TRANSITIONS`, `STATUS_CONFIG`, status helpers
- `shared/letterTypes.ts` — `LETTER_TYPE_CONFIG`, intake types
- `shared/pipeline.ts` — `PipelineContext`, `PipelineError`, pipeline-related types
- `shared/pricing.ts` — already partially exists, move remaining pricing types
- `shared/types.ts` — re-export barrel for backwards compatibility

---

### 8. `client/src/pages/DocumentAnalyzer.tsx` (1,101 lines)
**Problem:** File upload, analysis display, recommended actions, and the full page layout in one component.

**Refactor:**
- `components/analyzer/FileUploadZone.tsx`
- `components/analyzer/AnalysisResults.tsx`
- `components/analyzer/RecommendedActions.tsx`
- Keep `DocumentAnalyzer.tsx` as page-level composition

---

### Priority order I'd recommend:
1. **`email.ts`** — highest impact, cleanest split, no cross-dependencies
2. **`stripeWebhook.ts`** — event handlers are naturally independent
3. **`supabaseAuth.ts`** — security-critical code deserves clear boundaries
4. **`routers/letters.ts`** — tRPC routers merge trivially
5. **`shared/types.ts`** — barrel re-export makes it non-breaking
