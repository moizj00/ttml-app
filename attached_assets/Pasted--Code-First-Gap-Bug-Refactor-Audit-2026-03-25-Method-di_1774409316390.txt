## Verified facts from code

1. **Role model**: 4 roles exist in schema and route gating (`subscriber`, `employee`, `attorney`, `admin`).
2. **Affiliate mechanics**: employee discount codes, validation, usage counting, and 5% commission ledger entries are implemented in checkout + webhook paths.
3. **Submit UI gating exists**: subscriber submit page calls `billing.checkCanSubmit` and can block rendering.
4. **But submit API still runs independently**: `letters.submit` itself does not enforce entitlement before request creation + pipeline trigger.
5. **Review system currently allows attorney approvals**: review router mutations use `attorneyProcedure` (admin also allowed, but not admin-only).

## Gaps / errors vs requested target behavior

### A) Server-side entitlement gap at submission (Critical)
`letters.submit` creates letter + launches pipeline without server-side entitlement enforcement. UI precheck is not sufficient for policy integrity.

### B) First-letter-free path still active across billing + unlock logic (Critical)
`checkLetterSubmissionAllowed` grants `firstLetterFree`; `checkPaywallStatus` includes free state; `freeUnlock` transitions to `pending_review` without payment; DB/pipeline include previous free unlock semantics.

### C) Subscriber can see AI draft before final approval (Critical)
Subscriber flows render paywalled draft content (`LetterPaywall` receives draft text) and version access allows `ai_draft` during `generated_locked` flow.

### D) Final approval authority mismatch (Critical)
Requested model is admin-only final approval, but review actions (`approve/reject/requestChanges/saveEdit`) are under attorney-capable procedure/route model.

### E) Core status machine deviates from requested pending_review handoff (High)
Current lifecycle includes `generated_locked` paywall stage and optional auto-advance behavior for previously unlocked letters.

### F) Membership + annual plan policy mismatch (High)
Current plans are finite allowance-style (`lettersAllowed`) and legacy plan aliases; requested behavior specifies membership per-letter charges and annual 48-letter decrement model.

### G) Letter taxonomy mismatch (High)
Canonical code enums/config use current legal letter set (demand-letter, eviction-notice, consumer-complaint, etc.), not the requested 5-type set.

### H) Employee “view referred letters” capability missing (High)
Employee dashboard is affiliate-centric (code, earnings, payouts, commissions). No dedicated employee endpoint/view for referred subscribers’ letter list/details.

### I) Subscription discount persistence risk (Medium)
Referral coupon creation uses Stripe coupon `duration: "once"`; this may not provide recurring-cycle subscriber discount behavior.

### J) Broken/weak route consistency for employee links (Medium)
Webhook notifications/emails reference `/employee/dashboard` while client routes expose `/employee`, `/employee/referrals`, `/employee/earnings`.

### K) Semantic helper mismatch (Low)
`getEmployees()` returns both employee and admin roles, which can skew interpretation in consumers expecting employees-only.

### L) Product copy/flow mismatch around “attorney” vs “admin” review ownership (Low/Policy)
UI and route naming consistently use attorney review center; requested behavior says admin is sole reviewer/approver.

---

## Refactor opportunities

1. **Policy centralization**: single entitlement engine used by `letters.submit`, paywall checks, checkout creation, and webhook transitions.
2. **Authorization split**: separate “can review queue” from “can finalize approval” policies.
3. **Plan-policy unification**: one canonical map for prices, discounts, quota/decrement rules, and legacy alias translation.
4. **Route constants**: centralize route strings for emails/notifications/UI nav to prevent broken deep links.
5. **Business-contract tests**: entitlement, subscriber visibility boundaries, admin-only approval, referred-letter access, and annual decrement correctness.

---

## Suggested order
1. Enforce submit-time entitlement on server.
2. Remove free unlock/free first-letter policy paths.
3. Lock subscriber content to final-approved only.
4. Move final approval authority to admin-only.
5. Rebuild membership/annual commercial logic and taxonomy.
6. Add employee referred-letter visibility API + UI.
7. Normalize route constants and helper semantics.
