I checked the Google OAuth path specifically: /api/auth/google, /api/auth/callback, /api/auth/google/finalize, and the client login/signup pages that complete the flow.

What is good

The custom PKCE work is intentional, not random duct tape. The server generates its own verifier/challenge, stores the verifier in an HttpOnly cookie, and exchanges the auth code directly against Supabase’s token endpoint because exchangeCodeForSession() was unreliable in this SSR setup  ￼

The redirect handling is also reasonably careful. next is forced to a safe relative path, role is limited to subscriber or employee, and post-auth redirects are checked against role-allowed routes before use  ￼

What I’d flag
	1.	The flow is operationally brittle across environments.
getOriginUrl() only accepts the main production hosts plus certain Railway and Replit patterns. If you introduce another preview hostname or custom staging domain, auth links can silently fall back to https://www.talk-to-my-lawyer.com, which means OAuth or verification can bounce users to the wrong environment without much explanation  ￼
	2.	You still store OAuth tokens in localStorage after finalize.
Both login and signup finalize handlers write sb_access_token and sb_refresh_token into localStorage even though the server already set the sb_session HttpOnly cookie, and the tRPC client then sends that bearer token on every request when present  ￼  ￼  ￼

That is the biggest OAuth-specific security smell here. Cookie-first on the server reduces damage, but it does not erase it. Any XSS gets the token immediately, and stale token/header weirdness stays possible.
	3.	Google login and Google signup are inconsistent.
The login page clears old localStorage tokens before starting Google OAuth. The signup page does not. That asymmetry is exactly how “works in login, weird in signup” bugs are born  ￼  ￼
	4.	The PKCE verifier cookie is intentionally strict, but that makes the flow less forgiving.
/api/auth/google sets pkce_verifier with httpOnly, secure: true, sameSite: "lax", and the callback hard-fails if that cookie is missing. That is fine on correctly configured HTTPS production, but brittle on local dev, odd proxies, or privacy-heavy browser environments. The code even logs that exact failure mode  ￼
	5.	Boot-time env validation still misses a key auth dependency.
Google OAuth shares the same underlying auth stack as login/reset/verification, and those paths use the Supabase anon/public key. But startup validation does not require it, so auth can fail at runtime even when the app booted cleanly  ￼  ￼
	6.	This flow absolutely needs integration tests.
The callback path is custom and explicitly bypasses the higher-level Supabase client because of storage behavior in SSR. That may be the correct fix, but it is now a hand-built auth bridge. Those are famous for breaking after small dependency changes and then ruining everyone’s afternoon  ￼

My verdict

The Google OAuth implementation is thoughtful and more competent than most. But it is also custom, stateful, and sharp-edged. The main technical risk is not “this is obviously broken.” It is “this works until environment drift, browser behavior, or one library change nudges it sideways.”

Patch order I’d use

First, stop persisting OAuth access/refresh tokens in localStorage and go cookie-first only.

Second, make signup clear stale auth tokens before Google initiation just like login does.

Third, add the anon/public key to required env validation.

Fourth, add an end-to-end test matrix for:
	•	Google login via code flow
	•	Google signup via hash/implicit fallback
	•	admin Google login with 2FA
	•	preview/staging host redirect correctness
