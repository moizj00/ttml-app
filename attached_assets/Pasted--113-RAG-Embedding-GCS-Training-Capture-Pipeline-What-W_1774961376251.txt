#113 - RAG Embedding + GCS Training Capture Pipeline
What & Why
The database has pgvector, training_log, fine_tune_runs tables, a letter_versions.embedding column, and a match_letters() similarity function — all deployed directly to Supabase but with zero application code to use them. This task builds the application-layer code to:

Generate and store embeddings on letter approval (powering RAG retrieval for future drafts)
Capture training examples to Google Cloud Storage for future Vertex AI fine-tuning
Wire the RAG context into the drafting stage so the AI sees similar approved letters
Auto-trigger fine-tuning when enough examples accumulate
Done looks like
When an attorney approves a letter, the final version is automatically embedded (OpenAI text-embedding-3-small) and stored in letter_versions.embedding
A training example (intake + approved output) is appended to a JSONL file in the gs://ttml-training GCS bucket and logged in training_log
The drafting pipeline stage retrieves the top-3 similar approved letters via match_letters() and injects them as few-shot context
At 50+ training examples since the last fine-tune, a Vertex AI fine-tuning job is triggered automatically and tracked in fine_tune_runs
GCP config (project ID, bucket, region) is read from environment variables — no hardcoded values
Drizzle schema is updated to include training_log, fine_tune_runs, and letter_versions.embedding column definitions so the ORM knows about these DB objects
Out of scope
Admin UI for viewing/managing fine-tune runs (future work)
Switching the live pipeline to use fine-tuned models (requires separate evaluation)
Backfilling embeddings for any previously approved letters
Tasks
Drizzle schema sync — Add training_log and fine_tune_runs table definitions to drizzle/schema.ts, and add the embedding vector column to letter_versions. These objects already exist in the DB so no migration is needed — just the ORM type definitions.

Embedding service module — Create a module that generates embeddings via OpenAI text-embedding-3-small (1536 dims), stores them in letter_versions.embedding, and provides a findSimilarLetters() function that calls the existing match_letters() Supabase function.

Training capture module — Create a module that formats approved letter data as a training example (JSONL), appends it to a date-partitioned file in the GCS bucket, and logs the capture in training_log. Read GCS bucket name and GCP project from env vars.

Vertex AI fine-tune trigger — Create a module that checks count_training_examples_since_last_tune(), and when the threshold (50) is met, submits a Vertex AI supervised fine-tuning job and records it in fine_tune_runs.

Post-approval hook wiring — In the attorney approval flow (server/routers/review.ts), call the embedding service and training capture module after a letter is approved. These should be fire-and-forget (non-blocking) with error logging — a failure must never block the approval.

RAG injection into drafting — In the drafting pipeline stage, call findSimilarLetters() with the current letter's intake text, and inject the top results as few-shot examples in the drafting prompt.

Environment config — Create a .env.llm template file documenting the required GCP environment variables (project ID, region, bucket, credentials path). Add startup checks in the worker for GCS/Vertex availability with clear warnings if not configured.
