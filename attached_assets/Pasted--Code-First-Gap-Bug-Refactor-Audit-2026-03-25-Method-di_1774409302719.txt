## Verified implementation facts

1. **4 roles are implemented** (`subscriber`, `employee`, `attorney`, `admin`) in schema constants, DB enum, and route gating.
2. **Affiliate flow exists end-to-end in code paths**: employee code generation, checkout discount metadata, commission ledger writes, payout workflow.
3. **Review flow is attorney+admin capable**, not admin-only (review mutations use `attorneyProcedure`).
4. **Status machine is paywall-first** (`... -> generated_locked -> pending_review`) with additional states (`client_approval_pending`, `client_approved`, `sent`).

## Gaps / Errors vs requested behavior

### A) Entitlement enforcement is not guaranteed before generation (Critical)
`letters.submit` creates request and triggers pipeline execution path directly; it does not perform a strict single-letter-purchase/membership/annual-quota entitlement gate at mutation entry.

### B) Free-first-letter unlock remains active in backend and UI (Critical)
- Backend exposes `checkPaywallStatus` free-state and `freeUnlock` mutation.
- Paywall UI renders a prominent â€śSubmit for Free Reviewâ€ť CTA when eligible.

### C) Subscriber can access pre-approval AI content (Critical)
- Subscriber letter detail renders `LetterPaywall` with `draftContent`.
- Version access allows subscriber visibility of `ai_draft` for `generated_locked` letters.

### D) Final review authority mismatch (Critical)
Approval/reject/request-changes/save-edit are exposed through attorney-capable routes rather than admin-only workflow.

### E) Plan logic diverges from target commercial model (High)
Current plan model is fixed-plan/allowance based (`single_letter`, `monthly`, `yearly`, `lettersAllowed`) instead of:
- membership with per-letter metered charge while active,
- annual 48-included-letter decrement model.

### F) Letter taxonomy mismatch (High)
Current canonical letter types differ from requested list (e.g., includes `eviction-notice`, `employment-dispute`, `consumer-complaint`, `general-legal`; missing explicit `pre-litigation settlement` naming).

### G) Employee referred-letter visibility missing (High)
Employee affiliate router exposes code/earnings/commissions/payouts; no employee endpoint found for viewing referred subscribersâ€™ letters.

### H) Subscription discount persistence risk (Medium)
Referral coupon creation uses Stripe coupon `duration: "once"`; this can discount only first invoice on subscription-style purchases instead of persistent discounted recurring billing.

### I) Broken/invalid route in notifications (Medium)
Webhook-generated employee links use `/employee/dashboard`, while app router exposes `/employee` (plus `/employee/referrals`, `/employee/earnings`).

### J) Hardcoded jurisdiction option in submit form (Medium)
`SubmitLetter` imports full `US_STATES` list but currently renders only `California` in the state selector.

### K) Full draft can be revealed client-side from paywall component (High)
`LetterPaywall` stores full draft in DOM and toggles blur off via local UI state (`showFullPreview`), enabling unpaid full-text reading in-browser.

### L) Product messaging mismatch in public UI (Medium)
Public pages/components still advertise â€śfirst letter free,â€ť which conflicts with strict paid-before-review requirement.

### M) Naming/semantic smell in DB helper (Low)
`getEmployees()` returns both `employee` and `admin`, which can skew employee-specific analytics and admin screens if reused broadly.

---

## Refactor opportunities

1. **Single entitlement policy engine** shared by submit/paywall/checkout/webhook.
2. **Server-enforced content visibility policy** (never ship full ai_draft to subscriber clients before approval).
3. **Role-capability matrix** separating assignment from approval authority.
4. **Canonical plan/taxonomy registry** consumed by frontend, checkout, and backend validators.
5. **Route constants module** for all notification/email deep links.
6. **Business-contract tests** for: paid gating, admin-only approval, referred-letter visibility, quota decrement, discount persistence.

---

## Suggested execution order
1. Remove free unlock + enforce entitlement at submit.
2. Stop delivering full `ai_draft` to subscriber clients pre-approval.
3. Make final review actions admin-only.
4. Align pricing/plan/quota engine and letter taxonomy.
5. Add employee referred-letter visibility.
6. Fix route constants and state-selector/UI messaging mismatches.
