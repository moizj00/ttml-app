## Verified code facts

1. **4 roles exist and are route-gated** (`subscriber`, `employee`, `attorney`, `admin`).
2. **Affiliate mechanics exist**: employee code generation, code validation, 20% discount handling at checkout, 5% commission creation in webhook.
3. **Submit page has a UI gate** (`billing.checkCanSubmit`) and blocks the page when not eligible.
4. **Review actions are attorney-capable** (not admin-only) via `attorneyProcedure`.
5. **Paywall includes free-first-letter flow** in both backend (`freeUnlock`) and frontend CTA.

---

## Gaps / Bugs vs requested target behavior

### A) Backend entitlement enforcement is missing at submission mutation (Critical)
`client/src/pages/subscriber/SubmitLetter.tsx` checks entitlement in UI, but `server/routers.ts` `letters.submit` still allows direct submission + pipeline start without server-side entitlement check.

### B) Free-first-letter path conflicts with strict paid flow (Critical)
`LetterPaywall` renders “Your First Letter is Free” CTA and calls `billing.freeUnlock`; backend supports `checkPaywallStatus => free` and `freeUnlock` status transition.

### C) Subscriber can access pre-approval draft content (Critical)
`LetterDetail` passes `aiDraftVersion` into `LetterPaywall` and renders preview/download options for unreviewed content while status is `generated_locked`; this conflicts with “final content only after admin approval.”

### D) Final review authority is not admin-only (Critical)
`review.approve/reject/requestChanges/saveEdit` are guarded by `attorneyProcedure`, so attorneys can perform final actions when assigned.

### E) Role-area boundary blur in employee routes (High)
`/employee`, `/employee/referrals`, and `/employee/earnings` all render the same `AffiliateDashboard` component; there is no dedicated referred-letters experience despite “employees can view letters from referrals.”

### F) Missing employee referred-letter API (High)
Affiliate router exposes code/earnings/commissions/payout endpoints but no endpoint to list letters tied to referred subscribers.

### G) Commercial model divergence (High)
Code uses finite `lettersAllowed` plan model (`single_letter/monthly/yearly`) rather than the requested business rules (membership with per-letter billing and annual 48-letter decrement semantics).

### H) Canonical letter type mismatch (High)
`shared/types.ts` and submission enum use current technical values (`demand-letter`, `contract-breach`, etc.), not the requested business taxonomy.

### I) Subscription discount persistence risk (Medium)
Stripe coupons for referral codes are created with `duration: "once"`; this usually discounts only initial invoice for subscriptions, conflicting with “20% off purchases” interpretation.

### J) Route mismatch in employee notifications (Medium)
Webhook notifications use `/employee/dashboard`, while router exposes `/employee` and aliases without `/employee/dashboard` route.

### K) Naming/design smell in DB helper (Low)
`getEmployees()` returns both `employee` and `admin`, which can skew downstream assumptions/reports.

---

## Refactor opportunities

1. **Centralize entitlement policy** in one server service used by submit, paywall checks, checkout creation, and webhook status transitions.
2. **Admin-only approval policy module** separating assignment from approval authority.
3. **Single plan-policy map** (pricing + quota + decrement timing) shared across frontend/backend/webhook.
4. **Employee referral visibility module** (API + UI routes) separate from earnings ledger.
5. **Route constant registry** for server-generated links (emails/notifications) to avoid dead links.
6. **Business-contract tests** for entitlement, role authority, referral visibility, and post-approval content visibility.

---

## Suggested execution order
1. Enforce entitlement at `letters.submit` and remove/retire free unlock.
2. Enforce final-only subscriber content visibility.
3. Switch final review actions to admin-only.
4. Implement membership + annual quota policy engine.
5. Add employee referred-letter API/UI.
6. Normalize route constants and helper naming.
